# AI Reading-Companion Feed

A custom RSS feed generated by AI from books Martin is actually reading: one post per
chapter — a short reflection plus a pointer back to the chapter — consumed in FeedMe like
any other feed. Agreed 2026-07-16; the decision log at the bottom records what was chosen
and why. The implementation lives in its own repo (**`feedme-books`**, planned) — this doc
is the product-side spec.

## Goals & non-goals

**The feed exists to:**

- pace alongside a book Martin is currently reading — each post resurfaces one chapter's
  core idea and links to the chapter, turning the feed into a reading habit engine;
- let new books be added with one command: feed the script a book, it batch-generates all
  chapter posts at once;
- stay a **slow feed** — at most one post a day in drip mode, zero engagement mechanics —
  exactly FeedMe's anti-noise philosophy applied to reading.

**Non-goals:**

- **No user-facing "share a book" feature — rejected permanently** in its general form.
  Letting users submit arbitrary books means redistributing AI derivatives of commercial
  (often pirated) books at scale — a real copyright liability, not a hypothetical. If a
  user feature ever returns, it is scoped to open-licensed books only.
- **No FeedMe changes.** [backend.md](./backend.md) already defines a Source as any RSS
  endpoint, polled once. The generator emits a valid `feed.xml` at a public URL; the app
  consumes it like any blog. Until the backend lands (the app is frontend-only on mock
  data today), the feed is read in any standard RSS reader.
- **No authoritative summaries.** A post is a reflection and a pointer, never "everything
  in chapter 7" — see the post format contract below.

## How it works

Two separate concerns, deliberately split:

```
GENERATE (one-time per book, local)        RELEASE (ongoing, automated)
book config ──► fetch chapters ──►         queued posts ──► publish --next (daily cron)
Claude writes draft posts as files            │                    │
        │                                     │                    ▼
        ▼                                     │            feed.xml rebuilt + committed
Martin reviews: edit / delete / queue ────────┘            (raw GitHub URL serves it)
```

1. **Generate** — `generate <book>` fetches every chapter (raw markdown for book #1),
   calls Claude once per chapter (`claude-opus-4-8`, adaptive thinking; ~20 calls ≈ $1–2
   per book), and writes each post as a markdown file with `status: draft`. Idempotent:
   chapters whose post file exists are skipped, so deleting a bad draft and re-running
   regenerates just that one.
2. **Review** — the quality gate, and the hallucination guard: nothing publishes unseen.
   Martin reads the drafts, edits or deletes, then flips survivors to `status: queued`.
3. **Release** — only `released` posts enter `feed.xml`. Two modes:
   - `publish --all` — release every queued post at once;
   - `publish --next` — release the single oldest queued post; a **daily GitHub Action**
     runs this and commits only when the feed changed. The drip works by **rebuilding the
     XML** — the queue lives outside the feed as unreleased files. Never by future-dating
     items: pollers ingest whatever items exist in the XML regardless of `pubDate`.

The generator never runs in CI — no API key leaves Martin's machine; the Action only
promotes and rebuilds.

## Post format contract

- **Title:** `«Book» · Ch. N — «Chapter title»` (e.g. *Mostly Adequate · Ch. 5 — Coding by
  Composing*). The book name in every title is what makes one combined feed navigable:
  FeedMe's search/filter by book title pulls up exactly that book's chapters.
- **Body:** ~150–250 words — **one core idea** from the chapter, a short reflection on it,
  **one verbatim quote**, and the "read the chapter" link (the item's `<link>` points at
  the chapter's reader URL). One idea per post; depth lives in the book, not the feed.

## Planned repo layout (`feedme-books`)

Python — official `anthropic` SDK, `feedgen` for RSS 2.0, `httpx` for fetching. Same
language and parsing ecosystem as the planned backend ([backend.md](./backend.md) chose
Python for exactly this); `trafilatura` is available later for books that only exist as
messy HTML.

```
feedme-books/
  books/mostly-adequate.json     # title, author, license, chapters: [{n, title, sourceUrl, readerUrl}]
  posts/mostly-adequate/…md      # frontmatter: status draft → queued → released, releasedAt
  src/generate.py                # chapter markdown → Claude → draft post file (idempotent)
  src/publish.py                 # released posts → feed.xml; modes --all | --next
  .github/workflows/release.yml  # daily cron: publish --next, commit if changed
  feed.xml                       # https://raw.githubusercontent.com/MartCube/feedme-books/main/feed.xml
```

**First book:** the [Mostly Adequate Guide](https://mostly-adequate.gitbook.io/mostly-adequate-guide)
(CC BY-SA 4.0) — its chapters live as markdown on GitHub
([`MostlyAdequate/mostly-adequate-guide`](https://github.com/MostlyAdequate/mostly-adequate-guide),
ch01–ch13), so extraction is reading raw `.md` files while each post links to the gitbook
chapter page Martin actually reads.

## Decision log

- **Reading companion, not discovery roulette** _(2026-07-16)_ — posts follow the books
  Martin is actually reading, in chapter order. Sampling random chapters across a big
  catalog was rejected: posts about books you're not reading rarely convert to reading,
  and drift toward exactly the shallow scroll-content FeedMe exists to avoid.
- **Batch generation + separate release step** _(2026-07-16)_ — generating all posts at
  once (instead of a daily AI job) gives a free review gate, one-batch API cost, and no
  always-on generation infrastructure. Release cadence is an independent choice per
  publish run (all at once vs. one/day drip).
- **One combined feed** _(2026-07-16)_ — over one-feed-per-book. Simpler generator, one
  subscription; in-app search filters by the book name carried in every post title.
- **Separate public repo** _(2026-07-16)_ — the daily drip commits (~365/yr of
  "release: ch N") would pollute the app repo's history; a separate repo also gives the
  free raw-URL hosting for `feed.xml`.
- **Python** _(2026-07-16)_ — over Rust (no official Anthropic SDK, compile friction on a
  small text pipeline) and TypeScript. Aligns with the planned Django backend's language
  and parsing libraries.
- **Open-licensed books only; stage-3 user feature rejected** _(2026-07-16)_ — see
  non-goals. The public repo republishes AI-derived posts, so sources must be legally
  redistributable (Mostly Adequate is CC BY-SA 4.0; SICP, Pro Git, Eloquent JavaScript,
  the Rust Book are future candidates).
